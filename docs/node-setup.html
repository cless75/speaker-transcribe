Развёртывание ASR-узла
распознавание речи + диаризация

Пошаговая инструкция по установке узла транскрибации с диаризацией спикеров и пер-проектным voiceprint на новой Windows-машине. Узел работает как headless-обработчик: читает источники, делает ASR на GPU, складывает результаты в общий Hub.

Windows 10/11 NVIDIA CUDA 12.8 Python 3.11 faster-whisper + pyannote self-contained

iО документе

Этот файл лежит в репозитории и открывается локально (один HTML-файл, без внешних зависимостей). Он проведёт от чистой машины до работающего узла, который участвует в распределённой обработке записей.

Принцип: код публичный — данные приватные. В этой инструкции все токены, пути и имена — плейсхолдеры. Подставляйте свои значения локально. Никогда не коммитьте реальные секреты, голосовые отпечатки, транскрипты и имена в репозиторий. Голос = биометрические персональные данные; он живёт только в вашем приватном Hub. Подробнее — раздел Приватность.
Плейсхолдеры в командах: <...> — заменить на своё; {cache_root}, {hub_root} — корни путей вашей машины.

⚡Быстрый путь: установка через AI-агента

если на машине уже есть Claude Code или Codex

Если у вас установлен Claude Code или Codex (либо другой агент с доступом к терминалу) — не копируйте команды вручную. Делегируйте установку агенту: он сам адаптируется к вашему железу (GPU, диски, пути), обработает типовые затыки (ExecutionPolicy, PATH, cuDNN) и доложит результат каждого гейта. Дайте агенту такой промпт:

Установи speaker-transcribe на эту машину (Windows, NVIDIA CUDA).

1. Поставь Git (winget Git.Git) и Python 3.11 (winget Python.Python.3.11), если их нет.
2. Склонируй https://github.com/cless75/speaker-transcribe в C:\work\speaker-transcribe
3. Следуй docs/node-setup.html: venv в C:\work\venvs\asr; torch 2.11.0 + torchaudio 2.11.0
   из индекса https://download.pytorch.org/whl/cu128; затем pip install -r requirements.txt;
   nvidia-cudnn-cu12; ffmpeg (winget Gyan.FFmpeg).
4. Вызывай venv python напрямую (C:\work\venvs\asr\Scripts\python -m pip ...) — без активации,
   чтобы обойти PowerShell ExecutionPolicy.
5. Проверь гейты и доложи: nvidia-smi видит GPU; torch.cuda.is_available() == True;
   оба smoke-теста (PYTHONUTF8=1) зелёные.

Среда: только диск C:. GPU — внешняя NVIDIA RTX 3050 (eGPU). Движок CUDA-only.
Агент пройдёт те же шаги, что в разделах ниже, но без ручного копирования. Дальше (токены, конфиг, режимы) — там же. На macOS уточните агенту: CPU-профиль (int8), без CUDA/cuDNN, ffmpeg через brew (см. секцию macOS).

~2 мин · клонирование репозитория

1.1 · Установить Git (если не установлен)

:: проверить наличие
git --version
:: если «не является командой» — поставить через winget:
winget install --id Git.Git -e
После установки Git откройте новое окно терминала — PATH обновляется только для вновь запущенных процессов. То же правило позже для Python, cuDNN, ffmpeg.
Git pager включён по умолчанию: git log / git diff открывают постраничный просмотр (выход — клавиша q). Для скриптов и неинтерактивных сессий отключите его: git config --global core.pager cat (или разово git --no-pager <команда>).

1.2 · Клонировать репозиторий

git clone https://github.com/cless75/speaker-transcribe C:\work\speaker-transcribe
cd C:\work\speaker-transcribe

Репозиторий самодостаточен по коду — пути под машину задаются в конфиге (шаг 5), Python-файлы не редактируются.

2GPU и драйвер

первый аппаратный гейт

  1. Установить актуальный драйвер NVIDIA (Game Ready / Studio).
  2. Для eGPU: убедиться, что порт Thunderbolt включён в BIOS и устройство авторизовано в Windows.
  3. Проверить, что карта видна:
nvidia-smi
Гейт: вывод показывает вашу видеокарту и объём VRAM. Если карты нет — проблема на уровне драйвера/Thunderbolt, дальше идти бессмысленно.

3Python-окружение

~15–25 мин · venv + зависимости + cuDNN + ffmpeg

3.1 · Установить Python 3.11 (если нет) и создать venv

:: проверить наличие 3.11 (системный может быть другой версии)
py -3.11 --version
:: если нет — поставить, затем открыть НОВОЕ окно терминала:
winget install --id Python.Python.3.11 -e

:: создать venv
py -3.11 -m venv C:\work\venvs\asr

:: --- ВАРИАНТ A (рекомендуется): без активации, вызывать venv python напрямую ---
::     работает всегда, не зависит от политики выполнения скриптов
C:\work\venvs\asr\Scripts\python -m pip install -U pip wheel setuptools

:: --- ВАРИАНТ B: активировать venv, чтобы потом писать просто pip/python ---
::     PowerShell по умолчанию блокирует .ps1 → сначала разрешить для текущей сессии:
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
C:\work\venvs\asr\Scripts\Activate.ps1
::     в cmd.exe вместо двух строк выше: C:\work\venvs\asr\Scripts\activate.bat
Две частые ошибки на Windows:
• pip «не распознано» — venv не активирован; глобального pip в PATH обычно нет.
• «выполнение сценариев отключено в этой системе» — PowerShell блокирует Activate.ps1.
Простое решение для обоих: используйте Вариант A — вызывайте C:\work\venvs\asr\Scripts\python -m pip ... напрямую (далее в инструкции команды с pip подразумевают активированный venv либо этот полный путь).
Именно 3.11 — у pyannote/torch есть готовые бинарные колёса под него; на самых новых интерпретаторах pip может полезть собирать из исходников (медленно, часто падает на Windows).
Когда активация НЕ нужна: активация venv — это лишь удобство для интерактивных команд установки (чтобы писать короткое pip/python). Для запуска самого движка активация не требуется — интерпретатор берётся из явно заданного пути:
  • CLI / watcher — поле transcribe_python в конфиге (шаг 5);
  • PowerShell-обёртка — параметр -PythonBin;
  • режим агента (MCP) — переменная TRANSCRIPTION_PYTHON.
Поэтому автозапуск/планировщик/сервис работают без активации — указывайте полный путь к ...\venvs\asr\Scripts\python.exe.

3.2 · PyTorch под CUDA 12.8 (ставить отдельно от индекса PyTorch)

# сборки +cu128 не лежат на PyPI — нужен индекс PyTorch
pip install torch==2.11.0 torchaudio==2.11.0 --index-url https://download.pytorch.org/whl/cu128

3.3 · Остальные зависимости

pip install faster-whisper==1.2.1 ctranslate2==4.7.1 \
  "pyannote.audio==4.0.4" "huggingface_hub<1.0" \
  speechbrain==1.1.0 soundfile numpy psutil
# либо целиком из лок-файла (кроме torch — он из индекса cu128 выше):
# pip install -r requirements.lock.txt

3.4 · cuDNN 9 (для GPU)

faster-whisper/CTranslate2 на GPU требует cuDNN-DLL в PATH (иначе «Could not load cudnn…»). В активированном venv:

:: pip-пакет (проще всего — кладёт cuDNN в venv)
pip install nvidia-cudnn-cu12
:: либо системный cuDNN 9.x и добавить его bin в PATH

3.5 · ffmpeg (извлечение аудио из видео)

Обязателен для обработки видео и многих аудиоформатов. Установка на Windows:

:: через winget (явный id — просто "ffmpeg" может не найтись)
winget install --id Gyan.FFmpeg -e
:: после установки откройте НОВОЕ окно терминала (PATH), затем проверьте:
ffmpeg -version
Альтернативы: choco install ffmpeg или scoop install ffmpeg; либо вручную — скачать build с ffmpeg.org / gyan.dev, распаковать и добавить папку bin в PATH. Готово, если ffmpeg -version печатает версию.
На macOS — brew install ffmpeg (см. секцию macOS).
Гейт окружения:
C:\work\venvs\asr\Scripts\python -c "import torch; print(torch.cuda.is_available())"
# ожидается: True

4Токены

для чистого ASR + диаризации достаточно одного — HF_TOKEN

ТокенЗачемОбязателен
HF_TOKEN (Hugging Face)gated-модели pyannote (диаризация)Да
Google OAuthэкспорт/импорт Google DocsНет (для ASR не нужен)

4.1 · HF_TOKEN — получение

  1. Войти на huggingface.co.
  2. Принять условия gated-моделей под тем же аккаунтом (в браузере, кнопка «Agree»):
    • huggingface.co/pyannote/segmentation-3.0
    • huggingface.co/pyannote/speaker-diarization-3.1
  3. Settings → Access Tokens → New token, тип Read, скопировать hf_...

4.2 · HF_TOKEN — хранение (код читает HF_TOKEN / HUGGINGFACE_TOKEN)

# Способ A — CLI-логин (рекомендуется): пишет в %USERPROFILE%\.cache\huggingface\token
huggingface-cli login

# Способ B — переменная окружения
setx HF_TOKEN "hf_<ВАШ_ТОКЕН>"
Токен привязан к аккаунту, не к машине — на новом узле просто повторите логин.
Никогда не коммитьте токен в репозиторий и не кладите в Hub. В репозитории — только *.example с пустыми плейсхолдерами.

5Конфигурация узла

машинно-локальный конфиг из шаблона; декларативные источники → приёмники

Скопируйте шаблон в локальный конфиг (он gitignored) и подставьте свои пути:

copy config\node.example.json config\node.local.json
{
  "node": { "host_label": "<UNIQUE-NODE-NAME>",
            "capabilities": ["gpu-cuda","diarization","asr-fast"],
            "cache_root": "C:/work/cache" },
  "runtime": { "device": "cuda", "compute_type": "float16",
               "diarization_device": "cuda" },
  "transcribe_python": "C:/work/venvs/asr/Scripts/python.exe",
  "sources": [ { "root": "{hub_root}/_inbox", "route": "mapper", "claim": true } ],
  "outputs": {
    "transcript_dir": "{hub_root}/{pid}/sessions/{YYYY-MM}/{sid}/transcripts",
    "voiceprints": { "global_registry": "{hub_root}/_voiceprints",
                     "project_projection": "{hub_root}/{pid}/Speakers",
                     "local_cache": "{cache_root}/{pid}" }
  },
  "hub_root": "<GOOGLE_DRIVE>/Hub",
  "voiceprint_mode": "off",
  "enable_multi_machine": true
}

Создайте рабочие папки: локальный кэш, входящие, Heavy-зеркало:

mkdir C:\work\cache  C:\work\Heavy
voiceprint_mode: off — безопасный старт. Включайте пер-проектный voiceprint, когда настроите глобальный реестр голосов в Hub.

6Проверка и запуск

smoke-тесты → пробный прогон

6.1 · Smoke-тесты

Важно: задайте PYTHONUTF8=1 перед запуском — иначе вывод кириллицы/ символов падает с UnicodeEncodeError на системной кодировке Windows (cp1251).
set PYTHONUTF8=1
C:\work\venvs\asr\Scripts\python scripts\smoke_intake_tiering_a1.py
C:\work\venvs\asr\Scripts\python scripts\smoke_promote_archive_a1.py
Ожидается: оба теста — все PASS (например 19 pass / 0 fail и 15 pass / 0 fail).

6.2 · Пробный прогон узла

C:\work\venvs\asr\Scripts\python scripts\audio_inbox_watch.py --once --config config\node.local.json
Успех: файл заклеймлен, ASR прошёл на GPU, транскрипт и state.json появились в Hub с вашим host_label.
Ложный сбой: прогон может рапортовать exit 9 (teardown CUDA) при фактическом успехе. Судите по наличию transcript.md и phase=stdout_json ok, а не по коду возврата.

7Три режима использования

логика ASR общая — различается оркестрация и источник файлов

Движок используется тремя способами — спектр от оркестрации к простоте: через AI-агента, напрямую из командной строки или просто складывая файлы в локальную папку. Выбирайте по задаче.

A · Из агента (Claude Code / Cursor)

MCP-сервер даёт агенту инструменты транскрибации с preflight-проверкой среды; агент оркеструет полный пайплайн (входящие → ASR → хаб → карточки сессий / инсайты) и навыки вроде «обработай свежие записи».

Слой агента (MCP) и кластерный watcher — в дорожной карте репозитория (см. README → Status). Ядро движка ниже (режимы B и C) уже доступно.

B · Из командной строки — один файл

Прямой ASR одного файла; для скриптов и разовых прогонов.

# прямой ASR одного файла
python src\media_transcribe_cli.py --input <media> --output-dir <dir> \
  --model medium --speaker-mode on --timestamps both

# PowerShell-обёртка с явными параметрами прогона
pwsh scripts\run-media-transcribe-direct.ps1 -InputPath <media> `
  -RuntimeDevice cuda -Model medium -SpeakerMode on -ChunkMinutes 20

C · Локальная папка — ручной drop + пакетная обработка

Самый простой режим: складываете медиафайлы в локальную папку и запускаете пакетную обработку — транскрипты появляются в выходной папке. Без хаба, без мультиузла, без агента. Этот режим тоже конфигурируется — но локально: задаёте свои входную и выходную папки.

Конфигурация (локальный срез контракта «источник → приёмник»):

{
  "sources": [ { "root": "C:/work/inbox", "route": "folder", "claim": false } ],
  "outputs": { "transcript_dir": "C:/work/output/{name}" },
  "voiceprint_mode": "off"
  // никаких hub_root / enable_multi_machine — всё локально
}

Запуск (пакетно по всей папке):

# обработать все медиа во входной папке (PowerShell)
Get-ChildItem C:\work\inbox -Include *.m4a,*.mp4,*.wav -Recurse -File | ForEach-Object {
  python src\media_transcribe_cli.py --input $_.FullName --output-dir C:\work\output `
    --model medium --speaker-mode on
}
Подходит для разовой пакетной обработки своих записей без инфраструктуры: бросил файлы в папку → получил транскрипты. Папки — это и есть «конфигурация» локального режима.
Во всех режимах: PYTHONUTF8=1 для кириллицы в выводе; ASR — CUDA-only (fail-fast без GPU).

+Мультиузел (несколько машин)

Узлы координируются через Hub файлами .claim.json + heartbeat. Каждому узлу — свой уникальный host_label. Канонический результат (транскрипт + state) всегда в Hub, поэтому машины не зависят от локальных дисков друг друга.

Одна GPU на несколько машин (переключаемая карта). Если видеокарта одна и переезжает между машинами, запускайте watcher только там, где карта сейчас физически подключена. Узел без CUDA честно падает fail-fast и не «застолбит» файл.

🔒Приватность

Код публичный — данные приватные. Движок не собирает и не передаёт ваши данные. В репозиторий едут только код, схемы и *.example. Никогда не попадают в репозиторий и в Hub:
  • токены и секреты (credentials.local.json, OAuth-файлы);
  • голосовые отпечатки (_voiceprints/, profiles/*.json) — биометрические ПД;
  • транскрипты, аудио, имена участников, личные пути.

Убедитесь, что .gitignore покрывает: *.local.json, voiceprint-дерево, секреты, медиа и транскрипты.

🍎Вариант: macOS (Apple Silicon)

M-series · CUDA отсутствует → ASR на CPU (int8)

На Mac (M1–M4) нет NVIDIA/CUDA, поэтому движок работает в CPU-режиме. Ниже — дельта от Windows-инструкции; шаги по токенам (4) и идея конфигурации (5) те же.

Главное отличие: здесь нет CUDA/cuDNN/eGPU. Производительность ниже (CPU int8 ≈ 2× реального времени). Ускорители whisper.cpp Metal / MLX — опционально, вне этой инструкции.
